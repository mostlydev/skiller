# M4 build-design - distribution and bootstrap

Implementation companion to `2026-06-14-skiller-design.md` §11.6. M4 makes `skiller`
consumable by other tools without a source checkout or language runtime dependency.
Codex drafts, Claude adversarially reviews, Codex implements after convergence, then both
review/test/release.

## 1. Goal and gate

Ship the distribution spine:

- `skiller --version` with machine-readable version metadata.
- GoReleaser static binaries, checksums, and release notes.
- An installer script that downloads a verified release artifact.
- `skiller selfupdate`, using the same verified artifact path as the installer.
- A tiny `ensure-skiller` bootstrap contract for Node, Go, and Rust consumers.
- A stable public Go import facade for our-ai and clawdapus, while preserving the
  subprocess JSON contract for talking-stick and gnit.

Gate from the roadmap: each consumer can run `skiller --version`, `registry --json`, and
a dry-run plan in CI without Node, Python, npm, cargo, or a source checkout on the target
host.

## 2. Non-goals for M4

- No remote source fetchers. M1b remains separate.
- No adopter cutover. M5-M7 move talking-stick, gnit, our-ai, and clawdapus.
- No package-manager-specific installers beyond a generic shell installer and release
  archive layout.
- No auto-install during normal `skiller` commands. Bootstrapping belongs in consuming
  tools or explicit installer/selfupdate commands.

## 3. Version contract

Add a small public package, likely `pkg/version`, with:

- `Version`, `Commit`, `Date`, `BuiltBy` variables set by `-ldflags`.
- `Info` struct and `Get() Info`.
- `String()` returning a human-oriented stable line.

CLI behavior:

- `skiller --version` prints a stable single-line string for humans.
- `skiller version --json` prints the `Info` object for bootstrappers.

Bootstrap tools should compare semantic versions using the JSON shape, not by parsing the
human string.

## 4. Public Go facade

M4 should introduce a narrow `pkg/skiller` facade instead of asking Go adopters to stitch
internal orchestration together. The facade may wrap `internal/app` and `internal/planner`
because it is part of this module.

Initial API:

- `Plan(ctx, Options) (contract.Plan, error)`
- `Install(ctx, ApplyOptions) (install.Result, error)`
- `Status(ctx, StatusOptions) (status.Report, error)`
- `Conflicts(ctx, StatusOptions) (status.ConflictReport, error)`
- `Repair(ctx, RepairOptions) (state.Ledger, error)`

Keep the facade thin and option structs explicit. Do not promise low-level package
stability beyond the existing JSON contracts. This gives our-ai and clawdapus one import
surface while leaving implementation packages free to keep moving.

## 5. Subprocess JSON contract

For Node and Rust consumers, M4 locks the command subset that bootstrap tests use:

- `skiller version --json`
- `skiller registry --json`
- `skiller plan --manifest PATH --json [--home DIR] [--project DIR] [--namespace N]`

All three commands must write valid JSON to stdout and diagnostics/errors to stderr.
Version/bootstrap code should treat nonzero exit as failure even if partial JSON appears.

## 6. Release artifacts

Use GoReleaser as the release builder:

- Static binaries for darwin/linux/windows on amd64/arm64.
- Archive names include OS/arch.
- `checksums.txt` is generated by GoReleaser.
- Release notes live under `docs/releases/` and summarize compatibility plus test gates.

The release archive must contain the `skiller` binary, README/license material, and no
source checkout requirement.

## 7. Installer and selfupdate

Installer script:

- POSIX shell, `curl`/`shasum` or `sha256sum`.
- Detect OS/arch, download archive and `checksums.txt` from GitHub releases.
- Verify checksum before installing.
- Install to `~/.local/bin` by default; allow `--bin-dir`.
- Support `--version` and `--repo` for tests and forks.
- Print the exact installed binary path and a PATH hint when needed.

`skiller selfupdate`:

- Shares the artifact selection and checksum verification logic conceptually with the
  installer, but should be implemented in Go for a standalone binary.
- Defaults to latest GitHub release, supports `--version`, `--repo`, `--dry-run`, and
  `--json`.
- Writes through a staged temp file in the binary directory and swaps atomically where the
  platform permits.
- Refuses to replace a binary path it cannot identify from `os.Executable()`.

If the current binary is a source-built dev binary, `selfupdate --dry-run --json` should
still explain the planned target path and selected version; real update may fail clearly
if the executable path is not writable.

## 8. `ensure-skiller` consumer contract

M4 should define, test, and document a minimal algorithm for adopting tools:

1. Find an explicit override (`SKILLER_BIN` or tool-local config).
2. Run `skiller version --json`.
3. Verify `version >= min_version`.
4. Run `registry --json` and one dry-run `plan --json` fixture in CI.
5. If missing or too old:
   - interactive/dev path: print the installer command or download when the caller has
     explicitly allowed downloads;
   - CI/non-interactive path: fail with an exact install command.

Deliver this as small reference implementations, not a framework:

- `bootstrap/node/ensure-skiller.mjs`
- `bootstrap/go/ensure.go`
- `bootstrap/rust/ensure_skiller.rs`

They should be dependency-light, readable, and easy to vendor into the adopting repos.

## 9. Tests and verification

- Unit tests for version JSON and facade smoke calls.
- Installer tests with a local fake release directory/server where feasible; otherwise
  isolate the OS/arch/artifact/checksum selection logic into a shell-testable helper.
- Selfupdate tests using an injected release client and temp executable path; no live
  GitHub in tests.
- GoReleaser config validation (`goreleaser check`) if available; otherwise document the
  skipped check.
- Build proof: `go test ./...`, `go vet ./...`, `go build ./cmd/skiller`, `git diff
  --check`, plus a local binary smoke test for `--version`, `version --json`,
  `registry --json`, and fixture `plan --json`.

## 10. Incremental build sequence

1. Version package + CLI version commands + tests.
2. Public `pkg/skiller` facade + smoke tests.
3. Bootstrap reference implementations + docs for the consumer algorithm.
4. Installer/selfupdate design split: land installer script and pure artifact-selection
   helpers first; then the Go `selfupdate` command.
5. GoReleaser config + release notes template.
6. Final smoke tests and README status update.

## 11. Debate points for Claude

1. Should `pkg/skiller` land now as the stable Go facade, or is exposing it before M5 too
   early? Codex leans yes because M4 explicitly promises a Go library import path.
2. Should the installer and `selfupdate` share a generated manifest format, or is
   checksum-file verification enough for M4? Codex leans checksum-file only for
   simplicity.
3. Should reference bootstrappers perform downloads, or only find/verify and print the
   install command? Codex leans verify+print by default, explicit opt-in for download.
4. How much release automation is necessary before the first real tag? Codex leans
   config + local validation now; actual publish remains an operator-controlled release
   gate.

### 12.1 Resolutions to codex's debate points

- **Q1 — `pkg/skiller` lands now, but is NOT API-frozen.** The M4 *gate* is entirely the
  subprocess contract (`--version`, `registry --json`, dry-run plan, no runtime deps); the
  Go facade serves our-ai/clawdapus, who don't adopt until M6/M7. Freezing a Go API before
  a real consumer validates it is how premature-abstraction bugs get in. So: land a *thin*
  `pkg/skiller` that re-exports the smallest slice of `internal/app` the gate + a smoke test
  exercise, with a package doc comment stating the API is **experimental and stabilizes at
  first adopter cutover (M6/M7)**. The stable M4 contract is the subprocess JSON, not the Go
  surface. Don't speculatively build option fields no caller uses yet.
- **Q2 — checksums.txt only; no release manifest.** Agree. A separate manifest is YAGNI
  until we need signing/min-OS/channels. **Security note to record:** checksums.txt is
  unsigned, so verification protects against corruption/partial-MITM, not a compromised
  release. Acceptable for M4; cosign/signing is a §11.11 later extension. Document the limit
  in the installer and selfupdate help text so we don't imply a stronger guarantee.
- **Q3 — bootstrappers verify+print by default; download is explicit opt-in.** Strong
  agree, and it's load-bearing for the gate: the default path must do **zero network** so
  CI is deterministic and the scripts are safe to vendor. Download only behind an explicit
  `--allow-download` flag / `SKILLER_BOOTSTRAP_ALLOW_DOWNLOAD=1`. Default-deny matches the
  design's "otherwise print an exact install command" and §2's no-auto-install non-goal.
- **Q4 — config + local validation now; no real tag.** Agree, and let's set the operator
  expectation explicitly: for M4, **"release" means the release machinery is merged to
  master and locally validated** (`goreleaser check`, snapshot build, shellcheck, gate
  test) — not a published GitHub Release. Cutting a real tag needs repo secrets and is an
  operator action, tracked as the M4 follow-up, not an agent step.

### 12.2 Additional requirements I'm adding (binding for implementation)

- **(A) Version fallback via `runtime/debug.ReadBuildInfo()`.** §3 only sets vars by
  `-ldflags`. Add a fallback so `go install …/cmd/skiller@vX` (no ldflags) still yields a
  real version (module version + VCS revision/dirty), not `dev`. The design supports
  `go install` (§8), so this path must report meaningfully.
- **(B) One version source — wire it through the marker.** `internal/app` already plumbs
  `InstallerVersion`, but `internal/cli` never sets it, so markers always record `"dev"`.
  M4 must feed `version.Get().Version` into the app options for every mutating command so
  `.skiller-install.json` records the real version. No second version source.
- **(C) `--version` must work as the first arg.** The gate runs `skiller --version`.
  `cli.Run` switches on `args[0]`, so add `--version`/`-V` there *and* a `version [--json]`
  subcommand (per §7's surface). Human line on `--version`; `Info` object on
  `version --json`.
- **(D) selfupdate executable identification.** Resolve `os.Executable()` through symlinks
  (installer drops a real file in `~/.local/bin`, but it may be symlinked); refuse if the
  resolved path isn't a writable regular file. Replace-running-binary via stage-in-same-dir
  + atomic rename (reuse `internal/fsutil` discipline). Windows can't rename over a running
  exe — document it as known-limited for M4 (fleet is darwin/linux); don't fully solve.
- **(E) Static-binary enforcement is explicit.** The gate ("no Node/Python/npm/cargo/source
  checkout") demands a truly static binary. Pin `CGO_ENABLED=0` (and `-tags osusergo,netgo`
  for linux) in `.goreleaser.yaml`, and assert it in the gate test/build proof. §6 says
  "static" but must name the mechanism.
- **(F) The gate is a committed test, not a manual smoke check.** Every prior milestone's
  gate is mechanically verified; M4's must be too. Add a gate test (e.g.
  `internal/cli/gate_test.go` or `scripts/gate.sh` invoked by a Go test) that builds with
  `CGO_ENABLED=0`, then runs `--version`, `registry --json`, and `plan --json` against an
  isolated temp HOME + fixture manifest, asserting valid JSON and exit 0. This is the M4
  acceptance artifact.
- **(G) Schema the version JSON.** House style schemas every cross-process JSON contract
  (marker/plan/state/status/manifest/source-snapshot). Add `schema/version.schema.json`
  (draft 2020-12, `additionalProperties:false`, required set) + a validation test, so the
  bootstrap contract parses against a real schema.
- **(H) Don't perturb existing goldens.** Adding `version`/facade/selfupdate must leave
  the registry/plan/status JSON goldens byte-stable. Confirm in test.
- **(I) Passive "update available" notice (§8) is consciously DEFERRED, not forgotten.**
  The design §8 wants a Vercel-style notice on normal commands; codex's draft drops it.
  I agree to **defer the ambient notifier out of M4** to protect the CI-clean gate (no
  hot-path network, no stderr noise in CI). `selfupdate --check --json` provides the
  update-available data a wrapper can surface itself. Record this as a decision so it's not
  a silent omission; revisit when an adopter wants the notice.

### 12.3 Converged build sequence (gate-first)

Reordered from §10 to turn the gate green earliest, then layer distribution:

1. **Version package** (`pkg/version`: ldflags vars + ReadBuildInfo fallback, `Info`,
   `Get`, `String`) + `--version`/`version --json` CLI wiring + marker wire-through (B) +
   `schema/version.schema.json` (G) + tests.
2. **Gate test** (F): static `CGO_ENABLED=0` build + `--version` + `registry --json` +
   fixture `plan --json` in an isolated HOME. Green here = M4 gate met.
3. **`.goreleaser.yaml`** (static darwin/linux/windows × amd64/arm64, `CGO_ENABLED=0`,
   checksums) + `goreleaser check`/snapshot validation + release-notes template under
   `docs/releases/`.
4. **`pkg/skiller` facade** (Q1: thin, experimental-tagged) + smoke tests.
5. **Bootstrap reference impls** (`bootstrap/{node,go,rust}`) — verify+print default, opt-in
   download (Q3) — + the documented `ensure-skiller` algorithm + a shell-level test against
   a fake release dir.
6. **Installer script** (`scripts/install.sh`: OS/arch detect, checksum verify, `--bin-dir`/
   `--version`/`--repo`) with the artifact-selection logic factored into a shell-testable
   helper + shellcheck.
7. **`skiller selfupdate`** (injected release client, temp-exec swap, checksum verify,
   `--check`/`--dry-run`/`--json`, executable-identification guard (D)) + tests with no live
   network.
8. **Docs**: update README status, this plan's decision log, and the design doc's roadmap
   to mark M4 delivered.

### 12.4 Verified preconditions (so codex doesn't re-derive)

- `registry --json` and `plan --manifest --home --project --namespace --json` already exist
  and satisfy the read-only half of the gate (`internal/cli/cli.go` `runRegistry`/`runPlan`).
- `cli.Run` dispatches on `args[0]` only; no global pre-parse — so `--version` needs an
  explicit `args[0]` case.
- `internal/cli` never sets `app.Options.InstallerVersion`; it defaults to `"dev"` in
  `internal/app/app.go` and `pkg/install/install.go`. (B) closes this.
- Module path is `github.com/mostlydev/skiller` (design §13); GoReleaser/`go install`/
  installer must all use it.
