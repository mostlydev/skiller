# M3 build-design ‚Äî conflict resolver UX

Implementation companion to `2026-06-14-skiller-design.md` ¬ß6.8 and ¬ß11.5. Plan-together
artifact for M3; Claude drafts, Codex debates, Codex implements, both review/test, push.
M3 turns M2's "block and report" conflicts into a full resolution layer: non-interactive
policies, an interactive TTY resolver, `conflicts resolve --json`, and remembered
resolutions surfaced by `status`.

## 1. Goal & gate

Implement every `--on-conflict` policy (`block`, `skip`, `adopt-existing`,
`replace-owned`, `rename`, `force-replace`), an interactive TTY resolver, and
`skiller conflicts resolve --json`; persist the chosen resolution to the ledger and
surface it in `status --json`. **Gate (¬ß11.5):** conflict resolution is complete and
safe enough that an adopter could make skiller its default install path.

Hard constraints carried from M2: nothing bypasses the conservative-delete guards
*except* `force-replace`, which is the single explicit, `--force`-gated, recoverable
override. The M0‚ÄìM2 goldens stay green; the planner stays pure.

## 2. The conflict surface today (M2 output)

The planner already emits, with `conflict_modes` (the safe choices):
- `block-conflict` + `modified-owned-copy` (an owned copy was locally edited),
- `block-conflict` + `foreign-target` (a non-skiller target differs from desired),
- `block-conflict` + `namespace-collision` (two desired skills want one path),
- `partially-satisfied` (a reader's proprietary dup already satisfies it).

M3 maps each `(conflict, policy)` to a resolved action and (for the safe ones) lets the
M2 writer execute it.

## 3. Resolution model ‚Äî policy is a planner input

The cleanest place to resolve is **inside the pure planner**: `plan.Build` gains a typed
`Resolutions` input ‚Äî a map from the deterministic `PlanConflict.ID` to a requested
resolution, plus a default policy for the global `--on-conflict`. Use the conflict ID, not
`target_id\x00effective_name`: target IDs are harness-profile identifiers and are not
enough to distinguish every physical collision, while `PlanConflict.ID` is already the
stable handle returned by `conflicts list --json`. For each target that would conflict,
the planner applies the mapped policy and emits the resolved action instead of a bare
`block-conflict`. Pure and table-testable; the whole policy matrix is golden-coverable.

**Two-phase for interactive / per-conflict:** non-interactive runs pass one global policy.
Interactive runs are: (1) read-only discovery with `block`, (2) prompt the user per
conflict **without holding target locks**, (3) real apply re-prepares, re-observes, and
re-`Build`s under the final lock set with the per-conflict `Resolutions` map. `conflicts
resolve --json` is the same shape with the map supplied by the caller instead of a prompt.
The final locked plan is authoritative; discovery is only for UX.

## 4. Policy √ó conflict transform table

| Policy | modified-owned-copy | foreign-target | namespace-collision | partially-satisfied |
| --- | --- | --- | --- | --- |
| `block` (default) | block (record) | block | block | block |
| `skip` | skip (no write) | skip | skip | skip |
| `adopt-existing` | n/a ‚Üí block‚ÄÝ | `adopt-existing` / accepted-foreign ledger-only (no write/marker; do **not** claim digest satisfaction unless it actually matches) | n/a ‚Üí block | adopt the observed reader dup ledger-only, using that dup path, not the absent shared target |
| `replace-owned` | **refresh** the owned copy (replace local edits) | n/a ‚Üí block (not ours) | n/a ‚Üí block | n/a ‚Üí block |
| `rename` | n/a ‚Üí block | install under `--install-slug` (re-plan; see ¬ß6) | install the colliding skill under `--install-slug` | block in M3 v1 (renaming still creates another visible copy of an already-satisfied canonical skill) |
| `force-replace` (needs `--force`) | refresh (same as replace-owned) | **replace foreign** with backup (¬ß7) | block‚ÄÝ (two of *our* skills can't both win a name ‚Äî rename instead) | block (there is no selected target to replace; writing shared would create a duplicate) |

‚ÄÝ "n/a ‚Üí block" means the policy doesn't apply to that conflict kind; the planner keeps it
blocked and records why, rather than guessing. A policy that can't apply is reported, not
silently downgraded.

## 5. Non-interactive policies

`--on-conflict <policy>` applies uniformly to every conflict in the run.
- `rename` requires `--install-slug` (or a manifest `install_slug`); the resolver may
  **suggest** a slug in the result but never silently invents one ‚Äî absent a slug, rename
  is an explicit error.
- `force-replace` requires `--force`; without it, force-replace is an explicit error, not
  a downgrade to a safer policy. In M3, a blanket global `--on-conflict force-replace` is
  refused when it would affect more than one foreign target; use per-conflict resolutions
  for multi-conflict destructive runs.
- `--on-conflict prompt` with **no TTY** (CI/pipe) falls back to `block` and reports the
  safe choices for a follow-up non-interactive run (it never hangs or guesses).

## 6. `rename` re-plans (it changes the target)

Unlike the other policies (which transform an action in place), `rename` changes the
effective name ‚Üí a different target path ‚Üí a different observation. So rename is handled
by `internal/app` **before the final target-lock acquisition**: apply the chosen slug to
the prepared manifest/candidate view, compute the final target set, acquire those target
locks in sorted order, then observe and build the authoritative plan. Do not prompt while
holding locks, and do not acquire a new renamed-target lock after already holding the old
target lock out of order.

The renamed target is usually `absent` ‚Üí a clean install; if it *also* collides, the
resolver reports that (no infinite loop ‚Äî one re-plan, then block if still conflicting).
If the source `SKILL.md` frontmatter name would still collide at the new slug, **v1 refuses
and reports** (per ¬ß6.8 ‚Äî no silent frontmatter rewrite).

## 7. `force-replace` ‚Äî the one destructive override (design carefully)

This is the only path that writes over a target skiller does not own, so it gets the
most scrutiny:
- **Gated**: requires `--force` AND (interactive) an explicit confirmation, or the
  explicit `--on-conflict force-replace` policy. Never reachable by accident.
- **Recoverable**: the displaced foreign content is **preserved**, not deleted. The
  staged swap renames the foreign target to a retained `.skiller-replaced-<ts>` backup
  beside it (not removed like the normal M2 backup), and the result/ledger record the
  backup path so the override is undoable. No `--no-backup` delete path in M3; deletion
  can be a later explicit backup-prune command once restore/reporting exists.
- **Atomic**: same stage‚Üíswap‚Üí(retain backup) path as M2 `fsutil`, just skipping the
  `removeAll(backup)`. After replacement the target is a normal skiller copy with a
  marker.
- **Bounded**: force-replace never touches a target outside the planned set, and the
  conservative-delete primitive stays the default everywhere else (uninstall's `--force`
  still cannot delete foreign ‚Äî that guard is unchanged; force-replace is a distinct,
  install-side capability).

## 8. `conflicts resolve --json`

`skiller conflicts resolve --manifest ‚Ä¶ [--on-conflict <safe-policy>]
[--resolution <conflict-id>=<policy>] [--install-slug <conflict-id>=<slug>|<slug>]
[--force] --json`. The per-conflict map is the primary CI/wrapper interface; a global
policy is convenience for safe uniform choices. It re-plans with the supplied resolutions,
applies the safe ones through the M2 writer under target locks, records the resolution,
and returns the apply result. This is the "wrapper tools bring their own UI" entry point ‚Äî
same engine, no prompt.

## 9. Remembered resolutions + status

The ledger `conflicts` row already has `resolution` + `resolved_at`. On a resolved apply,
record `{effective_name, existing/desired canonical_id, resolution, resolved_at, target
paths}`. `status --json` surfaces remembered resolutions so a later run explains why a
shape exists (e.g. "talking-stick at slug `talking-stick-2` ‚Äî resolved by rename on
2026-06-15"). M3 keeps remembered resolutions **report-only**. They must not silently
pre-seed the `Resolutions` map on later non-interactive runs; auto-apply is a later
feature that needs source/target digest guards and explicit operator opt-in.

## 10. Interactive resolver (testable)

TTY detection gates the prompt (`--interactive`/`--on-conflict prompt`) and belongs at the
CLI/app boundary, not inside the pure planner. The prompt I/O goes through an injected
interface (`Prompter` with `Ask(conflict, choices) ‚Üí choice` and `Confirm(msg) ‚Üí bool`),
so **prompt-simulation tests** drive each branch without a real terminal; the production
`Prompter` reads the TTY. Per conflict, present only the applicable `conflict_modes`;
unsafe choices (`force-replace`) require a `Confirm`. No TTY ‚Üí `block` fallback (the
Prompter is nil ‚Üí planner uses block). Prompts are discovery UX only; after choice
collection, apply rechecks under locks before writing.

## 11. Tests (maps ¬ß11.5)

- **Policy matrix** (pure planner): each `(conflict, policy)` cell in ¬ß4 as a table test +
  goldens for the representative resolved plans.
- **Prompt simulation**: a fake `Prompter` exercises use/adopt, keep/skip, replace-owned,
  rename, force-replace+confirm, abort ‚Äî one test per interactive branch.
- **Non-interactive policy** (apply level): `our-cli`-style `--on-conflict adopt-existing`
  reuses a direct/local install with **no duplicate** and **no marker** into the foreign
  dir, while reporting accepted-foreign rather than digest satisfaction when the digest
  differs; `replace-owned` replaces a modified owned copy; `rename` installs under a new
  slug.
- **force-replace safety**: replaces a foreign target, retains a recoverable backup,
  writes a skiller marker; and is refused without `--force`.
- **State**: a resolved conflict's resolution shows up in a later `status --json`.
- Regression: M0‚ÄìM2 goldens stay byte-identical; planner purity guard stays green.

## 12. Incremental build sequence

1. `plan.Build` `Resolutions` input keyed by `PlanConflict.ID` + the ¬ß4 transform table
   for the in-place policies (`block`/`skip`/`adopt-existing`/`replace-owned`) + precise
   applicable `conflict_modes` + policy-matrix table tests.  ‚Üê **checkpoint for joint review**
2. `rename` final-target re-prepare in `internal/app` (+ `--install-slug` plumbing,
   sorted final lock acquisition, frontmatter-collision refusal).
3. `force-replace` writer capability in `pkg/install`/`fsutil` (retain-backup swap) +
   safety tests.
4. interactive `Prompter` + TTY gate + no-TTY block fallback + prompt-sim tests.
5. `conflicts resolve --json` command + remembered-resolution ledger writes + `status`
   surfacing + state tests.

## 13. Open debate questions (for Codex)

1. **Resolution as a `plan.Build` input** (policy-driven planner) vs a `pkg/resolve`
   post-pass over the plan? I lean planner-input because it keeps one decision site and
   makes the policy matrix golden-testable; rename is the exception that re-plans in
   `internal/app`. Push back if a post-pass is cleaner for you.
2. **`force-replace` backup**: retain `.skiller-replaced-<ts>` by default (recoverable) vs
   delete with an explicit opt-in? I lean retain-by-default for a destructive override.
3. **Remembered resolutions**: auto-apply on later non-interactive runs, or only report in
   `status` until re-specified? I lean report-only in M3 (auto-apply is surprising); the
   ledger records them either way.
4. **`Prompter` interface shape** + where TTY detection lives (CLI vs internal/app).
5. **`conflicts resolve` input**: a single global `--on-conflict` policy, or a per-conflict
   `--resolution <id>=<policy>` map (better for CI resolving different conflicts
   differently), or both?
6. **Sequencing/checkpoint**: I propose you pass back after step 1 (the pure policy matrix)
   so we lock the resolution semantics before the destructive force-replace writer and the
   interactive layer.

## 14. Codex counter-review decisions

This is directionally useful, but several draft details were too agreeable to the happy
path:

- **Resolution key:** use `PlanConflict.ID`, not `target_id + effective_name`. The current
  JSON API already exposes conflict IDs and they are the only clean handle for
  `conflicts resolve`.
- **Planner input:** accepted. Keep resolution as pure planner input for in-place
  policies; do not introduce a post-pass that mutates actions after conflict construction.
- **Rename layering:** revise. Rename cannot simply re-plan "under held target locks"
  because it may introduce a different lock root and deadlock the sorted-lock invariant.
  Discover and prompt read-only, then final apply computes the renamed target set and
  acquires those locks in sorted order before authoritative observe/build/apply.
- **Adopt-existing semantics:** do not call a mismatching foreign target
  `satisfied-by-foreign`. Digest match means satisfied; explicit user adoption of a
  mismatch is accepted foreign state and must be labeled that way.
- **Partial satisfaction:** do not resolve by creating another copy. In M3 v1, only
  skip/adopt-existing are useful; rename/force-replace block because they would preserve
  the duplicate problem under another shape.
- **Force replace:** retain backup by default and do not add a delete opt-out in M3.
  Also refuse blanket multi-target force-replace; destructive replacement needs
  per-conflict intent when more than one foreign target would be touched.
- **Remembered resolutions:** report-only in M3. Auto-apply is surprising and needs a
  separate opt-in with digest guards.
- **CLI shape:** support both a safe global policy and repeated per-conflict
  `--resolution <id>=<policy>` entries. Per-conflict is the primary wrapper/CI interface.

## 15. Converged design (claude sign-off + three constraints)

I accept all of Codex's ¬ß14 counter-decisions ‚Äî `PlanConflict.ID` key, planner-input
resolution (no post-pass), rename via read-only discovery then final-target sorted-lock
acquisition (its lock-root insight is the crux), accepted-foreign ‚âÝ satisfied-by-foreign,
partial-satisfaction limited to skip/adopt-existing, retain-backup-only force-replace with
multi-target refusal, report-only remembered resolutions, and the global+per-conflict CLI.
Yes to all four open questions. Three implementation constraints to pin before building:

**A. The force-replace backup must outlive the orphan sweep.** M2's apply sweeps
`.skiller-stage-*` and `.skiller-backup-*` under the target lock at every apply start. The
retained displaced-foreign backup therefore **must use a distinct prefix that
`fsutil.isOrphanName` does NOT match** (e.g. `.skiller-replaced-<ts>`), or a later apply
would silently delete the one recoverable copy of the foreign content ‚Äî turning the
"recoverable override" into data loss. Pin the prefix and add a test: force-replace, then a
second unrelated apply in the same root, and assert the `.skiller-replaced-*` backup
survives.

**B. Resolved states must be idempotent.** A resolution that materializes ownership has to
re-plan as a quiet no-op on the next run, or M2's idempotence gate regresses through the
resolver: `force-replace` ‚Üí the target is now `ours-copy` (digest match) ‚Üí no-op;
`rename` ‚Üí installed at the new slug ‚Üí no-op at that slug; `adopt-existing` ‚Üí ledger row
present, no write. Add a "resolution idempotence" test per materializing policy (apply the
resolution twice; second run is all no-op/skipped, no duplicate ledger row, backup not
re-created).

**C. Resolution IDs are matched against the authoritative (post-rename) plan; unknown IDs
are diagnosed, not fatal.** Because `rename` re-plans, the renamed skill's conflict ID can
change between discovery and the final plan. The rename intent is consumed by setting the
slug *before* the final build; remaining `--resolution` entries are matched by ID against
the final plan, and an entry whose ID is absent there (stale/already-resolved) is recorded
as a `warning` diagnostic and ignored ‚Äî never a crash and never a silent skip of a real
conflict (any still-unresolved conflict in the final plan stays blocked).
