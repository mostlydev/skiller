# M2 build-design ‚Äî writer, state writes, and conformance

Implementation companion to `2026-06-14-skiller-design.md` ¬ß6.5, ¬ß6.7, ¬ß11.4, ¬ß12.
Plan-together artifact for M2; Claude drafts, Codex debates, Codex builds, both
review/test, push. M2 is the **first mutating milestone** ‚Äî correctness and safety
dominate over feature breadth.

## 1. Goal & gate

Build the writer and make it safe: `pkg/install` (the only writer), `pkg/prune`,
`pkg/state` write/repair, `internal/lock`, `internal/fsutil`; the mutating CLI commands
`install`, `uninstall`, `sync`, `cleanup-duplicates`; legacy adoption adapters. Default
`--on-conflict block`. Every mutating command still emits the same `plan --json` under
`--dry-run`.

**Gate (¬ß11.4):** a full fixture run proves **idempotence**, **no shared-target
duplicates**, **no foreign clobber**, **safe uninstall**, and **deterministic JSON
status** across all four tool shapes. The M0/M1 goldens stay green; M2 adds apply-result
goldens and a conformance suite replaying talking-stick/gnit/our-ai/clawdapus scenarios.

## 2. The Apply pipeline (TOCTOU-safe)

The read-only path is `internal/app.Plan` (resolve ‚Üí observe ‚Üí pure plan). The mutating
path adds locks around it and an executor after it. Ordering is dictated by ¬ß6.5
(lock-before-observe) and ¬ß6.7 (state lock discipline):

```
internal/app.Apply(opts) (Result, error):
  reg, manifest, snapshots  := load + resolve sources        // impure, no target writes
  if opts.DryRun { return internal/app.Plan(opts.PlanOptions) } // no target locks, no writes
  candidates := target.Resolve(...)                          // pure
  lockIDs    := sortedCanonicalLockIDs(candidates)           // deterministic order (¬ß6.5)
  locks      := lock.AcquireTargets(lockIDs)                 // BEFORE observe; held through apply
  defer locks.Release()
  world      := observe.Observe(candidates, ...)             // under lock -> no TOCTOU
  p          := plan.Build(...)                              // pure, fresh under lock
  results    := install.Apply(p, applyOpts)                  // the ONLY writer
  state.Commit(ledgerUpdatesFrom(results))                   // takes state.json.lock AFTER target locks held
  return results
```

Key invariants, enforced and tested:
- Target write locks are acquired **before** `observe` and held through apply (no other
  process can mutate a root between the snapshot and the write).
- Multiple target roots ‚Üí locks acquired in **deterministic sorted canonical lock-id
  order** (no lock-ordering deadlock between two concurrent skiller runs).
- `state.json.lock` and per-source cache locks are **short-lived** and **never held while
  waiting for a target lock**. The state lock is taken only after target locks are held,
  for the ledger read-modify-write, then released.
- A real write (re)builds its plan **under the lock**. `--dry-run` remains a read-only
  command and delegates to the normal M1 plan path without taking target locks; it must be
  byte-identical to `skiller plan --json`, not a lock-contending pseudo-write. The user
  still gets a snapshot plan, and the real write still rechecks under lock.

## 3. internal/lock

File-based advisory locks. Three lock classes, one discipline:
- **target lock**: one per target root (`lockID = "target:" + shortHash(root)`, already
  the plan's `lock_id`). Acquired in sorted order, held through apply.
- **state lock**: a single global `state.json.lock`, short transactions only.
- **source-cache lock**: per `source_key` (M1b/M2 forward ‚Äî defined now, used when remote
  fetch lands). Released before any target lock is acquired.

Implementation decision (Codex counter-review): use `github.com/gofrs/flock` behind a
tiny `internal/lock` wrapper. This is the safety core and Windows correctness matters
more than avoiding one small dependency; do not hand-roll `LockFileEx` in M2 unless the
library proves insufficient. Lock files live under the state dir
(`<state>/locks/<encoded-id>.lock`), not inside target roots (so a lock never pollutes a
skill dir or its digest). The JSON `lock_id` remains human-readable (`target:<hash>`),
but the filename must be path-safe on Windows (colon is illegal), so `internal/lock`
encodes logical ids before touching the filesystem.

## 4. internal/fsutil ‚Äî atomicity & rollback

The hard safety core. All writes stage under the target's parent directory, so the stage
is on the **same filesystem** as the target by construction, then atomically `rename`:
- **copy install**: stage content in `<target-parent>/.skiller-stage-<rand>/`, compute
  the install digest (marker excluded), write `.skiller-install.json` into the stage,
  then `rename(stage, target)` only when the target is absent. If target exists and is
  owned+replaceable: `rename(target, <backup>)`, `rename(stage, target)`,
  `removeAll(backup)`; on failure, `rename(backup, target)` rollback. **Never** an
  in-place partial copy, and never rely on `rename` overwriting an existing directory
  (that is not portable, especially on Windows).
- **link install**: create `symlink(managedSource, <target-parent>/.skiller-link-<rand>)`;
  if target is absent, rename the temp link into place; if replacing an owned target, use
  the same backup/swap/rollback sequence as copy. On symlink failure
  (EPERM/unsupported FS) ‚Üí **fall back to copy** and record `mode.effective=copy`,
  `requested=link`, `fallback_applied=true` in the result.
- **same-FS guard**: staging in the target parent should make cross-device rename
  impossible. Keep an EXDEV check anyway and fail loudly rather than silently doing a
  non-atomic move.
- **durability**: best-effort fsync file contents and parent directory metadata on Unix
  before reporting success; document platform limits on Windows rather than pretending
  `rename` alone is crash-proof.

Determinism for tests: randomness in stage names comes from a seam (injected suffix) so
apply-result goldens stay stable.

## 5. pkg/install ‚Äî execute a Plan

`install.Apply(plan, opts) ‚Üí []ActionResult`. It is the **sole filesystem writer**, not a
pure function. It must not re-classify ownership or make policy decisions; it executes
each action's `planned_writes` through `fsutil` and records realized outcomes (including
fallbacks). One case per action verb already in the plan:
- `install-link` / `install-copy` ‚Üí stage+rename (+marker for copy).
- `install-extra` ‚Üí file copy (+ sidecar marker if `owned_marker`).
- `refresh` ‚Üí replace owned target (staged swap), rewrite marker.
- `adopt-existing` ‚Üí **ledger-only**: no filesystem write, no marker into a foreign dir.
- `no-op` ‚Üí nothing (idempotent quiet path).
- `block-conflict` / `partially-satisfied` ‚Üí **never executed**; surfaced as `blocked`
  results, recorded in state, non-zero advisory in the result.

`ActionResult` carries the realized status (`installed|updated|skipped|blocked|failed|
dry-run` ‚Ä¶), requested/effective mode, fallback information, and the actual writes
performed. The ledger update is derived by a small explicit reducer from `ActionResult`;
do not reuse the wire result struct as the ledger mutation shape.

## 6. Conservative delete & pkg/prune

- **Conservative delete (¬ß6.5)**: remove a copy only if marker `owner` matches **and** the
  live digest still equals `installed_digest_at_install`, unless `--force`. A symlink is
  removable only if its realpath is the managed source. Never touch a dir we don't own.
- **pkg/prune**: port talking-stick's **symlink-only** duplicate cleanup ‚Äî remove only
  symlinks resolving to the managed source; preserve foreign symlinks and unmanaged
  copies. `cleanup-duplicates` uses realpath/marker proof, never name-matching alone.
- **uninstall**: honors the Option-B rule ‚Äî single-target uninstall leaves shared
  `~/.agents/skills` and hints; shared removed only with `--shared`/`--all`.

## 7. pkg/state write/repair

- **Commit**: read-modify-write the single JSON ledger behind `state.json.lock`, write to
  a temp file, `rename` over `state.json` (never in-place). Update **after** successful
  apply (a blocked/failed action records a conflict row, not an install row). Records the
  latest known view (not an event log), per the M1 `state.Load` schema.
- **repair**: `state doctor`/`repair` rebuilds the ledger from manifests, registry roots,
  markers, legacy adapters, and live observation when the ledger is missing/corrupt
  (M1 already returns an empty rebuildable ledger; M2 makes the rebuild real).
- **Single ownership truth on adoption (¬ß6.7)**: when materializing ownership of an
  `ours-legacy` target (marker-upgrade / replace-as-owned), remove the recognized legacy
  marker (`.gnit-skill-managed`, `.our-managed.json`) **only after** the skiller marker /
  replacement is written successfully. Lineage is kept in the ledger, not the stale file.
  Applies only to `ours-legacy`; `foreign` markers are never touched.

## 8. Legacy adoption adapters (¬ß11.4)

Data-driven via `data/markers.toml` (already classifies ours-legacy vs foreign). M2 adds
the **adoption actions** for the ours-legacy rows: talking-stick bundled-source symlinks +
old proprietary duplicates; gnit `.gnit-skill-managed`; our-ai `.our-managed.json`
(installers `our`/`our-ai`, managed roots under `~/.local/share/our*`); clawdapus
`~/.claw/skill-installed` as a host-self-skill adoption **signal only** (never touches
runtime surface markdown). These produce `adopt-existing` (ledger-only) or
`refresh`/replace-as-owned (with marker cleanup) per the ¬ß6.7 table.

## 9. Conformance suite (the gate)

Replay each tool's real install behavior as fixture-home tests that **apply** and assert:
- **talking-stick**: install/uninstall/update-migration; shared symlink + claude copy;
  duplicate cleanup removes only symlinks-to-source, preserves copies; idempotent re-run
  is a quiet no-op.
- **gnit**: embedded managed-source materialization; `.gnit-skill-managed` adoption;
  marker read-error tolerance.
- **our-ai**: canonical IDs, install slugs, tool-provided skills, stale-vs-modified copy
  (stale ‚Üí refresh, modified ‚Üí preserve unless `--force`).
- **clawdapus**: `~/.claude/skills/clawdapus-cli` + `~/.agents/skills/clawdapus-cli`
  adoption without touching runtime markdown.
- **coexistence**: a direct-CLI install at the desired target ‚Üí `satisfied-by-foreign` or
  block, **never** a duplicate; `--on-conflict adopt-existing` writes **no** marker into
  the foreign dir and does not mutate it (ledger-only).
- **idempotence/determinism**: apply twice ‚Üí identical end state + quiet second run;
  `status --json` deterministic across all four shapes.

## 10. CLI

`install`, `uninstall`, `sync`, `cleanup-duplicates` ‚Äî each with `--dry-run`/`plan`
returning the same `Plan`. Default `--on-conflict block` (record conflict, refuse unsafe
mutation; full resolver UX is M3). `install` accepts a manifest or a single source spec;
`sync --manifest` prunes marker-owned skills no longer declared, preserving foreign +
locally-modified targets.

## 11. Incremental build sequence (keep tests green each commit)

1. `internal/lock` (+ logical-id filename encoding, cross-process exclusivity, and
   deadlock-order test: two goroutines, sorted acquisition).
2. `internal/fsutil` (atomic copy/replace/rollback, no rename-over-existing-dir
   assumption, symlink+fallback; unit tests with injected stage-suffix seam).
3. `pkg/install` apply for the clean cases (absent ‚Üí install-link/copy/extra, no-op).
4. `internal/app.Apply` lock‚Üíobserve‚Üíplan‚Üíapply‚Üístate pipeline + `install` CLI +
   `--dry-run` parity (dry-run plan == M1 plan).  ‚Üê **suggested checkpoint for joint review**
5. `pkg/state` Commit + ledger update after apply; `state` doctor/repair.
6. refresh/replace-owned, conservative delete, `uninstall`, three-way digest drift.
7. `pkg/prune` + `cleanup-duplicates`; `sync` prune semantics.
8. legacy adoption adapters + single-ownership marker cleanup.
9. conformance suite (¬ß9) + apply-result goldens; final gate run.

## 12. Open debate questions (for Codex)

1. **Lock impl**: tiny internal `syscall.Flock` wrapper (no dep) vs `gofrs/flock`?
   Codex counter: use `gofrs/flock` behind `internal/lock`; Windows correctness beats
   shaving a dependency in the mutating safety core.
2. **Apply re-plans under lock** vs accepts a pre-built plan: real writes re-plan under
   lock; `--dry-run` delegates to read-only `Plan` and takes no target locks.
3. **Result vs ledger source of truth**: `ActionResult` drives both the result JSON and
   the ledger update, but through an explicit reducer; keep the wire result shape
   separate from the ledger mutation shape.
4. **fsutil dir-replace strategy**: rename-to-backup + swap + rollback, explicitly never
   rename over an existing directory. Windows rename-over-dir is the gotcha to design
   around now.
5. **Conformance fixtures**: hand-authored fixture-homes in-repo, or extract/port
   talking-stick's actual test fixtures? I lean hand-authored minimal fixtures that encode
   the same assertions, to avoid coupling to talking-stick's TS test internals.
6. **Checkpoint cadence**: I propose you pass back after step 4 (apply pipeline + install
   clean cases) so we review the lock/atomicity core before the long tail, then again
   before the conformance gate. Smaller check-ins welcome.
7. **New deps**: M2 may want `gofrs/flock` (see Q1). Anything else (e.g. a richer test
   helper)? Keep the dependency surface minimal per the elegance directive.

## 13. Codex counter-review summary

The draft is directionally right, but the mutating milestone needs sharper contracts than
M0/M1. Decisions proposed above:

- **Use `gofrs/flock` behind `internal/lock`** instead of hand-rolling platform locks.
  This is a small dependency in exchange for a tested Windows path. The wrapper keeps
  replacement possible later.
- **Encode lock filenames.** Existing plan lock ids contain `:` (`target:<hash>`), which
  is not a valid Windows filename. Keep logical ids in JSON; encode before creating lock
  files.
- **Dry-run stays read-only.** Real writes re-plan under locks; `install --dry-run` /
  `sync --dry-run` must be target-lock-free and byte-identical to `skiller plan`.
- **`install.Apply` is not pure.** It is the only writer and must not reclassify or decide
  policy, but it does perform I/O and can realize link‚Üícopy fallback. Results are reduced
  into ledger updates by a separate explicit mapper.
- **No rename-over-existing-dir assumption.** The fsutil contract is backup/swap/rollback
  only, staged under the target parent, with EXDEV checks and best-effort fsync.

Non-blocking but important: the checkpoint after step 4 should include a real
cross-process lock test and a crash/rollback simulation for copy replacement before
moving into state repair, prune, or legacy adapters.

## 14. Converged design (claude sign-off + two additions)

I accept all five of Codex's safety decisions ‚Äî `gofrs/flock` behind `internal/lock`,
lock-filename encoding, dry-run staying read-only and byte-identical to `plan`,
`install.Apply` as the non-pure sole writer with an explicit `ActionResult`‚Üíledger
reducer, and the no-rename-over-dir / backup-swap / EXDEV / fsync `fsutil` contract ‚Äî and
its step-4 checkpoint (cross-process lock + rollback simulation). Two safety properties
the draft left implicit and that a mutating milestone must state explicitly:

**A. Crash / partial-apply recovery model.** A single `Apply` performs N independent
per-target actions; each is individually atomic (stage+rename), but the apply as a whole
is **not** a cross-root transaction (three separate roots cannot be swapped atomically).
The recovery contract instead is **idempotent self-healing**, with the filesystem as the
source of truth and the ledger as a rebuildable cache:
- The ledger is committed **once, after** all actions. A crash mid-apply leaves some
  targets written and the ledger un-updated ‚Äî but a re-run `observe`s the real FS, sees
  the already-written targets as `ours-symlink`/`ours-copy` (realpath/marker), plans them
  as `no-op`, completes the remainder, then commits. No duplicates, no double-write.
- **Orphan sweep:** a crash can leave `.skiller-stage-<rand>` / `.skiller-backup-<rand>`
  siblings in a target parent. At the start of every apply, **under the target lock**,
  sweep stale skiller stage/backup dirs for that root before staging. (Under the lock so a
  concurrent run's in-flight stage is never swept.) This is the only filesystem cleanup
  the recovery needs; everything else is reconciled by observe.
- Net invariant: re-running any interrupted `install`/`sync` converges to the same end
  state as one clean run ‚Äî which is exactly what the ¬ß11.4 idempotence gate already asserts,
  now also covering the crash path.

**B. Lock-contention policy.** Target-lock acquisition **blocks with a bounded timeout**
(default a few seconds, `--lock-timeout` to override), not an unbounded hang and not a
fail-fast. On timeout the command exits with a clear, non-zero `another skiller run holds
<root> (lock <id>)` error and performs **no** partial work (it has not observed or written
anything yet ‚Äî locks are acquired before observe). `state.json.lock` uses the same bounded
wait for its short transaction. This keeps two concurrent runs safe (sorted order prevents
deadlock; the timeout prevents a stuck process from wedging the fleet) without silently
proceeding on a contended target.
