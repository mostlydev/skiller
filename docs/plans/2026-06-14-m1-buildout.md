# M1 build-design — pure planner, registry, and state reads

Implementation companion to `2026-06-14-skiller-design.md` §5 and §11.3. This is the
design-and-debate artifact for M1; once converged it guides Codex's implementation.
Claude drafted it; Codex debates and implements; both review/test; release.

## 1. Goal & gate (recap)

Build the read-only spine as the real `pkg/` library: `pkg/registry`, `pkg/manifest`,
`pkg/source` (file resolver only), `pkg/observe`, **pure** `pkg/plan`, `pkg/state`
(reads/rebuild), `pkg/status`. CLI gains `status` and `conflicts list` alongside
`plan`/`registry`, all read/plan only. **No writes, no mutating command.**

**Gate:** `skiller plan --json` represents every adopter's current install intent — and
an existing direct-CLI install at the desired path is reused (`satisfied-by-foreign`),
never duplicated. The seven committed M0 golden plans + schema validation are the
**regression oracle**: the M1 refactor is behavior-preserving for plan output, so on the
clean fixture homes the goldens stay byte-identical. A golden diff is a red flag, not an
expected outcome.

## 2. Package decomposition (internal monolith → pkg DAG)

M0 put everything in `internal/planner/plan.go` (814 lines), mixing source resolution,
target resolution, FS observation, classification, and plan assembly. M1 splits it along
the design's observe/plan/apply spine and promotes the stable, importable units to
`pkg/` (per §5 — the package boundary *is* the product; Go adopters import these).

Proposed import DAG (arrows = "depends on"; acyclic):

```
              pkg/plan  ───────────────┐  (pure: no os, no io/ioutil, no net)
             /   │   │    \            │
   pkg/manifest pkg/registry pkg/source pkg/observe
          │          │          │       │
          └──────────┴──── pkg/target ─┘
                              │
                     internal/{fsutil,digest,lock} (impure leaf helpers)

   pkg/status ── pkg/state ── pkg/observe ── pkg/registry
   pkg/state  ── (reads schema/state.json ledger; no writes in M1)
```

- **Type ownership is distributed** (idiomatic Go): each package owns its primary type —
  `manifest.Manifest`, `registry.Catalog`/`Harness`/`MarkerRule`, `source.Snapshot`,
  `target.Ref`/`CandidateSet`, `observe.WorldState`/`RawObservation`,
  `plan.Plan`/`Action`/`Conflict`. `pkg/plan` sits at the top and consumes the rest, so
  there is no cycle. The current `internal/contract` is dissolved into these owners
  rather than kept as a god-package.
- **`internal/` keeps the impure leaf helpers** (`fsutil` atomic ops for M2, `digest`,
  `lock`) and the CLI wiring. Leaves have no skiller-domain imports.
- `data/*.toml` stay embedded via the root `embedded.go` (`go:embed`); `pkg/registry`
  reads them through the existing strict typed loader (unknown-field rejection stays).

## 3. The observe/plan purity split (the core of M1)

Today `classifyTarget` does both the FS read *and* the desired-vs-actual decision. M1
separates them so `pkg/plan` is a pure function and the whole §6.7 decision table is
table-testable without a filesystem.

**`pkg/observe` records raw, source-agnostic facts.** It does not know what is "desired":

```go
type RawObservation struct {
    Path          string
    Exists        bool
    IsSymlink     bool
    SymlinkTarget string          // EvalSymlinks result, if a symlink
    Digest        string          // content digest, installer metadata excluded
    SkillerMarker *MarkerFacts    // parsed .skiller-install.json (owner, canonical_id, installed_digest_at_install), if installer==skiller
    LegacyMarker  *LegacyFacts    // matched data/markers.toml row: owner_label, class, marker_path
    OwnershipTest *TestResult     // result of a named non-in-dir predicate (e.g. clawdapus global marker)
    Err           string
}

type WorldState struct {
    Observed map[string]RawObservation // keyed by absolute candidate path
}

func Observe(targets []TargetRef, reg registry.Catalog, home string) (WorldState, error) // impure
```

**`pkg/plan` classifies and decides, purely.** It compares each `RawObservation` against
the desired skill/source to assign the ownership *class* and `digest_match`, then applies
the decision table:

```go
func Build(in Inputs) Plan   // pure: Inputs{Manifest, Registry, Snapshots, WorldState, Options}
```

So `ownership.Class` (ours-symlink/ours-copy/ours-legacy/foreign-known/foreign-unmanaged/
satisfied-by-foreign/partially-satisfied/absent) is *derived in plan* from raw facts +
desired identity, not produced by the scanner. This resolves the "pure planner that still
scans the FS" inconsistency the design called out, and is the step M0 deferred.

## 4. Target resolution is pure and shared

Both observe (which paths to scan) and plan (which targets to act on) need the same
candidate set. Make it one pure function so they cannot drift:

```go
// pkg/target
func Resolve(m manifest.Manifest, reg registry.Catalog, opts Options) CandidateSet
```

`target.Ref` carries the resolved root/path, kind, scope, readers, lock id, and the
proprietary duplicate roots to survey for that target group (per §6.2 — readers' shared
+ proprietary roots, plus ledger-remembered locations, **not** every root on disk). The
orchestrator calls `target.Resolve`, feeds the refs to `observe.Observe`, then calls
`plan.Build`.

This package split is not bikeshedding. If `TargetRef` lives in `pkg/plan`,
`pkg/observe` must import `pkg/plan` to accept the refs, while `pkg/plan` imports
`pkg/observe` to consume `WorldState` — a direct Go import cycle. `pkg/target` is the
neutral shared package that keeps the boundary pure and compileable.

## 5. Orchestration

A thin facade composes resolve → observe → plan so the CLI stays dumb:

```go
// internal/app for M1; promote a pkg/skiller facade only after the public API stabilizes.
func Plan(opts Options) (plan.Plan, error) {
    reg := registry.Load()
    m := manifest.Load(opts.ManifestPath)
    snaps := source.ResolveAll(m, opts)              // impure: file resolver
    ledger := state.Load(opts.StateDir)              // impure: read-only
    targets := target.Resolve(m, reg, opts.withLedger(ledger))
    world := observe.Observe(targets, reg, opts.Home) // impure
    return plan.Build(plan.Inputs{m, reg, snaps, world, opts}), nil // pure
}
```

Keep the lower-level `pkg/*` packages importable in M1, but do **not** freeze a public
high-level orchestration facade yet. M1b source fetchers and M2 writes will almost
certainly change orchestration options; an `internal/app` facade lets the CLI stay thin
without making that churn public API.

## 6. pkg/source (file resolver)

Lift `resolveLocalSource` into `pkg/source` producing `source.Snapshot` (the M0
`SourceSnapshot` shape, already schema-locked). M1 ships only the `file` fetcher;
`source_key`, `canonical_uri`, digest, and frontmatter plumb through exactly as today.
Git/http fetchers are M1b — the `Fetcher` interface is defined now so M1b is additive.

## 7. pkg/state (read & rebuild, no writes)

`state.Load(dir) → (Ledger, error)` parses `state.json` against the locked
`state.schema.json`; a missing or corrupt ledger yields an empty ledger plus a rebuild
hint (not a fatal error). The ledger feeds two consumers:
- **plan**: remembered install locations expand the duplicate-survey set beyond registry
  defaults (so a prior install in a non-default root is still de-duplicated).
- **status**: the "what we think is installed" side of the reconciliation.

No write path in M1. `state doctor`/`repair` and the write path land with the M2 writer.

## 8. pkg/status + conflicts CLI

- `status.Report(...)` reconciles the ledger with live `Observe` of registry+ledger
  roots: live facts win; stale ledger rows surface as `orphaned`/`not-seen`;
  `satisfied-by-foreign` is reported distinctly from `installed` (update-propagation
  caveat, §6.7). Output locked by `status.schema.json`.
- `skiller status --json` and `skiller conflicts list --json`. In read-only M1,
  `conflicts list` surfaces the conflicts a fresh `plan` would record plus any persisted
  ledger conflicts; both are already represented in the plan/state schemas.

## 9. Test plan (maps §11.3)

- **Purity guard**: a test (or `go vet`-style check) that `pkg/plan` imports neither `os`
  nor `net` — enforces the pure boundary mechanically.
- **Fixture-home**: target resolution, shared-target grouping, namespace collision,
  manifest default precedence, `install_slug`/frontmatter collisions, registry validation.
- **State read/rebuild**: recognized foreign installs, legacy markers, direct-CLI
  installs, duplicate prevention, stale ledger entries; corrupt-ledger rebuild.
- **Digest-match**: unmarked direct-CLI install at desired path → `satisfied-by-foreign`,
  no marker written (read-only anyway).
- **Golden plans**: the seven M0 goldens must stay green through the refactor
  (behavior-preserving), and `status`/`conflicts` get their own goldens.

## 10. Open debate questions (for Codex)

1. **Promote to `pkg/` now?** I say yes (per §5/§11.3 — the boundary is the product, and
   moving it later is costlier). Counter if you'd rather keep `internal/` until a Go
   adopter actually imports it in M5.
2. **Distributed type ownership vs a central `pkg/contract`?** I propose distributed
   (plan at the DAG top), with a neutral `pkg/target` for shared target refs. Push back
   if you foresee another import cycle.
3. **`RawObservation` (observe) vs `ObservedOwnership` (plan-derived) split** — agree the
   scanner stays source-agnostic and classification moves to plan?
4. **Orchestrator home: `pkg/skiller` facade vs `internal/app`?** A `pkg/` facade is
   nicer for Go adopters but commits to a public surface earlier. Codex leans
   `internal/app` for M1, with importable lower-level `pkg/*` as the public contract.
5. **`conflicts list` source in read-only mode** — fresh-plan conflicts ∪ ledger
   conflicts, or plan-only for M1? Codex leans union when `--manifest` is supplied;
   without a manifest, ledger-only is the only meaningful read-only source.
6. **Refactor sequencing** — one big restructure commit, or incremental (extract
   registry/manifest/source first, then the observe/plan split, then state/status)? I lean
   incremental so each step keeps `go test ./...` + goldens green.

## 11. Codex counter-review

Decision proposals after reviewing the draft:

1. **Promote domain packages now, but not the facade.** `pkg/registry`, `pkg/manifest`,
   `pkg/source`, `pkg/target`, `pkg/observe`, `pkg/plan`, `pkg/state`, and `pkg/status`
   should exist in M1 because those are the library boundaries adopters will eventually
   use. Keep the orchestration entry point under `internal/app` until M1b/M2 settle
   remote fetching and writes.
2. **Add `pkg/target`; do not put target refs in `pkg/plan`.** The original draft's
   `pkg/plan.ResolveTargets` creates an import cycle once `observe.Observe` accepts
   `TargetRef` and `plan.Build` consumes `observe.WorldState`. `pkg/target` owns
   `Ref`, `CandidateSet`, and pure target expansion from manifest+registry+ledger hints.
3. **Raw observation boundary is right.** `pkg/observe` may parse marker files and run
   named ownership tests because those are source-agnostic facts. It must not decide
   whether a marker is "ours" for the current desired skill; that belongs in `pkg/plan`.
4. **Conflicts list should be source-aware.** With `--manifest`, return fresh plan
   conflicts union persisted ledger conflicts, deduped by conflict id and marked with
   provenance (`live-plan`, `ledger`, or `both`). Without `--manifest`, report
   ledger-only conflicts rather than inventing desired state.
5. **Refactor incrementally.** Do not do one big package move. Sequence:
   registry/manifest strict loaders; `pkg/target`; `pkg/source` file resolver;
   `pkg/observe` raw world with an adapter that keeps goldens stable; pure `pkg/plan`;
   state/status/conflicts CLI. Every commit keeps `go test ./...`, schema validation, and
   the M0 goldens green.

## 12. Converged design (claude + codex) — decision log

Debate closed; all six §10 questions resolved. This block is authoritative for
implementation.

1. **Q1 promote to `pkg/` now → YES**, domain packages only (`registry`, `manifest`,
   `source`, `target`, `observe`, `plan`, `state`, `status`). Orchestration stays in
   `internal/app`; no public `pkg/skiller` facade until M1b/M2 settle (Q4).
2. **Q2 type ownership → distributed**, plus a neutral **`pkg/target`** owning `Ref` /
   `CandidateSet` / pure expansion. Codex correctly caught that putting `TargetRef` in
   `pkg/plan` cycles with `pkg/observe`; `pkg/target` is the fix. DAG: `target →
   {manifest, registry}`; `observe → {target, registry}`; `plan → {manifest, registry,
   source, target, observe}`; `status → {state, observe, target, registry}` — acyclic.
3. **Q3 observe/plan boundary → confirmed.** `pkg/observe` records *facts* (exists,
   symlink target, digest, parsed `.skiller-install.json` fields, matched
   `data/markers.toml` row, named ownership-test results). It must **not** decide whether
   a marker is "ours" for the desired skill; the `owner`/`canonical_id`/`digest` *compare*
   and the resulting ownership class live in pure `pkg/plan`.
4. **Q4 orchestrator → `internal/app`** (deferred public facade), per above.
5. **Q5 `conflicts list` → source-aware.** With `--manifest`: fresh-plan conflicts ∪
   ledger conflicts, deduped by conflict id, each tagged provenance `live-plan | ledger |
   both`. Without `--manifest`: ledger-only (never invent desired state).
6. **Q6 sequencing → incremental**, per §11.5, goldens + schema tests green every commit.

**Purity is mechanically enforced, two parts:**
- **Hoist environment reads out of `pkg/target`/`pkg/plan`.** The only non-FS impurity in
  today's target resolution is `targetRoot`'s `os.Getenv(h.EnvHome)`
  (`CLAUDE_CONFIG_DIR`/`CODEX_HOME`/`GROK_HOME`). `internal/app` resolves env + home up
  front and passes a resolved snapshot (or pre-resolved roots) into `target.Resolve`, so
  `pkg/target` and `pkg/plan` read no process state.
- **Purity-guard test.** A `pkg/plan` test parses its own package's source with
  `go/parser` and asserts no direct `os`/`net` import — the minimal elegant check; deeper
  transitive purity is guaranteed by the acyclic DAG (plan's only deps are pure domain
  packages + `target`). Goldens stay byte-identical because plan re-derives the same
  ownership class/`digest_match` from the same raw facts + desired identity.

step; release when the M1 gate holds.
